# Архитектура Market Pain Miner

Дата: 2026-05-21

Market Pain Miner для WB/Ozon — локальная исследовательская система, которая
берет Telegram-экспорт, анализирует его и выдает человеку наглядный отчет:
что найдено, откуда взялись выводы, какие гипотезы появились и что проверить
дальше.

## Главная Идея

Проект не равен одной продуктовой гипотезе. `Stock Location Mismatch`,
manager lead radar, review helper и другие идеи — это результаты анализа.

Главный продуктовый контур:

```text
Telegram result.json
  -> импорт в SQLite
  -> классификация сообщений
  -> кластеры болей и вопросов
  -> готовые решения и конкуренты
  -> opportunity cards
  -> локальный multi-page сайт для человека
```

CLI остается движком обработки. Локальный сайт становится основным результатом
пилотного прогона, потому что без наглядности трудно удержать в голове связи
между сообщениями, выводами и гипотезами.

## Операционная Система Проекта

Проект использует идею starter-репозитория как “рабочую память”, а не как
готовый app scaffold:

| Элемент | Роль |
|---|---|
| `AGENTS.md` | постоянные правила для Codex в этом репозитории |
| `CODEX_AUTOPILOT.md` | протокол первого запуска и продолжения работы |
| `.business/` | приватный бизнес-контекст, аудитория, экономика, рынок |
| `skills/` | локальные навыки для повторяемых задач |
| `prompts/` | готовые промпты |
| `playbooks/` | короткие runbook-процессы |
| `plans/` | планы работ и валидаций |
| `retrospectives/` | короткая память по сессиям |
| `app/` | Python-код продукта |
| `data/` | локальные приватные данные и отчеты, игнорируются git |

## Слои Приложения

### 1. Импорт

`app/importers/telegram.py` читает Telegram Desktop `result.json`, нормализует
формат сообщений и возвращает внутренние dataclass-модели.

### 2. Хранение

`app/storage/sqlite.py` хранит:

- `chats` — metadata чатов;
- `messages` — сырые и нормализованные сообщения;
- `classifier_runs` — запуски классификатора;
- `message_labels` — автоматические и ручные метки.

SQLite выбран как первый слой хранения: локально, просто, тестируемо.

### 3. Классификация

`app/classifiers/` содержит rule-based классификатор. Он раскладывает сообщения
по категориям:

- `pain`;
- `question`;
- `solution_ad`;
- `tool_mention`;
- `case`;
- `insight`;
- `offtopic`.

Классификатор должен оставаться заменяемым: позже за тем же контрактом можно
подключить ручную разметку, локальные правила или LLM.

### 4. Кластеры

`app/clusters/` группирует боли и вопросы по явным problem markers, темам и
уникальным evidence keys. Кластер всегда должен показывать, какие message IDs
поддерживают вывод.

### 5. Решения И Конкуренты

`app/solutions/` извлекает упоминания ботов, сервисов, таблиц, курсов,
расширений, parser/tools и рекламных сообщений. В публичном выводе URL и
handles заменяются aliases.

### 6. Гипотезы

`app/opportunities/` превращает кластеры и решения в opportunity cards:

- проблема;
- сегмент;
- частота;
- текущий workaround;
- готовые решения;
- первый MVP;
- причина платить;
- сложность;
- риск;
- score;
- evidence.

### 7. Отчеты

`app/cli.py` дает команды:

- `import`;
- `classify`;
- `stats`;
- `summary`;
- `review`;
- `clusters`;
- `solutions`;
- `opportunities`;
- `report`.
- `site`.

`site` генерирует multi-page static site в `data/reports/`: `index`,
`for-you`, `people`, `tools`, `insights`, `niches` и `data/*.json`.

`report` остается быстрым single-page HTML-отчетом. Полноценный dashboard можно
делать позже, когда понятно, какие блоки сайта реально нужны.

## Модель Приватности

Приватность — часть архитектуры:

- `data/exports/`, `data/db/`, `data/reports/` игнорируются git;
- `.business/` содержит приватный бизнес-контекст;
- вывод по умолчанию не показывает реальные имена, handles, URL, user IDs,
  названия приватных чатов и сырые цитаты;
- `--raw-local` разрешен только для локальной отладки;
- в tracked-документах можно хранить только агрегаты, aliases и
  анонимизированные выводы.

## Текущий Архитектурный Выбор

Сейчас важнее не расширять классификатор бесконечно, а сделать результат
понятным человеку. Поэтому ближайший фокус:

1. сохранить надежный CLI-пайплайн;
2. генерировать локальный сайт после анализа любого чата;
3. показывать цепочку “данные -> кластеры -> решения -> гипотезы”;
4. сохранять приватность по умолчанию.

## Ближайшая Эволюция

- улучшить качество rule-based классификации по реальным пилотам;
- добавить FTS-поиск, когда появится потребность искать по базе;
- хранить версии HTML/Markdown отчетов и static sites в ignored `data/reports/`;
- позже добавить локальный dashboard поверх той же SQLite-базы;
- подключать API/LLM только после того, как визуальный исследовательский цикл
  докажет пользу.
